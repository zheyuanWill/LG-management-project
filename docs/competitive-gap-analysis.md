## 〇、定位与目标用户

### 公司背景

**上海鲁顾船舶服务有限公司（LG Marine Services）** 是一家成立于 2007 年的第三方海事服务公司，总部位于上海。公司核心业务包括：

| 业务线 | 说明 |
|--------|------|
| **修船/改装监造 (Repair & Conversion Supervision)** | 受船东委托，作为第三方驻场监督，把控船厂修理/改装项目的范围、质量、进度与交付 |
| **船舶经纪 (S&P / Newbuilding Broker)** | 二手船买卖撮合、新造船项目对接，提供市场信息与决策支持 |
| **售前检验 (Pre-sale Inspection & Condition Survey)** | 二手船交易前的船况评估与验船 |
| **新造船咨询与监造人力 (Newbuilding Consultation)** | 新造船项目咨询、驻厂监造团队管理 |
| **备件供应 (Spare Parts Supply)** | 为船东/项目提供船用备件供货支持 |
| **船厂代理 (Shipyard Agent)** | 在中国主要修/造船厂拥有合作网络（北/中/南），为国际船东对接船厂资源 |

> 来源：[lgmarineservices.com](https://lgmarineservices.com/)

### 系统定位

**LG Management 是 LG Marine Services 的内部项目管理与业务运营系统。** 它不是一个卖给船东的 SaaS 产品（至少在初期不是），而是帮助 LG 自身团队高效管理**多客户、多船厂、多业务线**的日常运营工具。其核心价值在于"监督方"视角——代表船东对船厂的工作进行范围控制、进度跟踪、变更索赔、验收取证、成本归集与结算对账。

| 维度 | 说明 |
|------|------|
| **目标用户角色** | Superintendent（驻场维修监督）、项目经理（PM）、技术采购/询价员（Procurement）、财务结算（Finance）、HSE/QA 巡检员、公司管理层（项目组合与业务概览） |
| **典型场景** | ① 修理/坞修监造：受船东委托驻场监督船厂执行，管理工作包、变更、验收、结算；② 备件供应：询价比价、采购、库存、交付；③ 船舶经纪：项目/交易跟踪（当前系统以①②为主） |
| **对协同方** | 向下对船厂/分包商/供应商：发出规格书 → 收集报价 → 下达工作包 → 监督执行 → 验收签字；向上对船东客户：进度汇报、费用预警、结项报告 |
| **多项目管理** | LG 同时服务多个船东客户、对接多家船厂，系统需要支持**跨客户、跨船厂的项目组合视图** |
| **关键 KPI** | ① 减少漏收索赔与变更遗漏（直接影响 LG 的项目利润）；② 缩短结算对账周期；③ 提升在修进度可见性（让船东客户实时可查）；④ 降低返工率与争议纠纷；⑤ 合规文档一次沉淀、随时可审；⑥ LG 自身的业务报表（按客户/船厂/业务线的收入与成本分析） |

> **与"船东自有系统"的区别**：DNV ShipManager、AMOS、VesselMan 等竞品面向的是**船东/船管公司内部团队**，管理自有船队的维修计划。LG 的场景不同——LG 作为第三方服务商，同时为多个船东客户提供监造服务，因此系统需要额外的**客户管理、多项目组合视图、LG 自身的利润核算**等能力。
>
> **与"修船厂侧 ERP"的区别**：船厂侧系统关注内部排产、工种派工、工时报工、车间级调度等"施工执行"环节（典型代表：摩杰斯修船 ERP）。LG Management 关注的是"第三方监督"视角——确认范围有没有做完、质量是否达标、变更是否合理、费用是否对得上。二者互补但出发点不同，本文在对比时会注明各竞品的侧重方向。

## 三、缺失功能分析与优先级

### 🔴 P0 — LG 监造业务核心工作流，不做就无法形成闭环

#### 1. 工作包分解与跟踪（Work Package Breakdown & Tracking）

**这是监督方的核心工作单元。** Superintendent 到厂后的第一件事就是把规格书拆成一个个可跟踪的工作包，然后逐项确认范围、跟踪进度、验收签字。

```
缺什么:
- 工作包模型: 包号、关联项目、所属系统/舱室/设备、工作描述、验收标准
- 工作包状态: 未启动 → 进行中 → 待验收 → 已验收 → 关闭（可返工）
- 与船厂工单的映射（可选）: 记录船厂侧的工单号，便于进度对照
- 工量/工时记录: 监督方记录实际消耗，或导入船厂提供的工时数据，用于结算对账
- 工作包级别的成本归集: 每个工作包的预算 vs 实际成本
```

**建议优先级**: 🔴 最高 — 没有工作包粒度的跟踪，"进度透明"无从谈起

#### 2. 变更单 / 索赔管理（Variation Order / Change Order）

**修理项目中，变更单往往是关键利润来源之一（比例因项目规模和类型而异）。** 船进坞后发现的额外问题、船东临时加装需求、船厂索赔，都通过变更单管理。漏记一条变更单 = 直接少收一笔钱。

```
缺什么:
- 变更单模型: 变更号、关联项目/工作包、发起方(船厂/船东/监督方)、原因分类、
  变更内容描述、金额影响(增/减)、审批状态
- 审批流: 监督方确认 → 内部审批 → 船东确认 → 纳入合同补充协议
- 与原合同/报价的关联: 合同总金额自动更新，原始报价 vs 最终结算可追溯
- 变更追溯时间线: 所有变更的完整历史（谁发起、谁审批、什么时候确认）
- 证据附件: 照片、船厂报告、检验报告等支撑材料
```

**建议优先级**: 🔴 最高 — 直接关系到 LG 的项目利润与争议防控

#### 3. 验收记录与证据链（Acceptance & Evidence Trail）

**监督方的核心交付物之一是"验收证据"——证明船厂做了什么、做到什么标准、谁签字确认的。** 一旦发生质保争议或保险索赔，证据链是唯一依据。

```
缺什么:
- 验收记录模型: 关联工作包、检查项清单、检验标准、实际值、合格/不合格
- 现场取证: 照片/视频 + 时间戳 + GPS（移动端拍摄自动附加）
- 签字确认: 监督方签字、船厂代表签字（电子签名或手写拍照）
- 缺陷清单 (Deficiency List): 不合格项登记 → 整改跟踪 → 复检关闭
- 可追溯导出: 按工作包导出完整验收报告（PDF），含所有照片、签字记录、检查结果
```

**建议优先级**: 🔴 最高 — 验收证据链是 LG 向船东客户交付的核心输出，也是商业谈判/争议解决的基础

#### 4. 结算对账（Settlement Reconciliation）

**当前结项管理只有粗粒度的收入/成本汇总，缺少按工作包/合同条款逐项对账的能力。**

```
缺什么:
- 按工作包对账: 每个工作包的合同价 vs 实际发生 vs 变更调整
- 对账单生成: 汇总全部工作包 + 变更单，生成可发给船厂/船东的对账单
- 争议标记: 对账过程中双方有异议的条目可标记、备注、跟踪解决状态
- 发票关联: 对账确认后关联发票，跟踪开票/收款状态
```

**建议优先级**: 🔴 最高 — 结算效率直接影响 LG 的现金流回收周期

#### 5. 报表与文档导出（PDF/Excel）

**已有 Celery 任务存根但完全未实现。** 对于监督方而言，输出标准化的文档（报价单、合同、验收报告、对账单、结项报告）是日常工作的基本需求。

```
缺什么 (已有存根，需实现):
- 报价单 PDF（含逐项明细 + 条款）
- 合同 PDF
- 验收报告 PDF（含照片证据链）
- 变更单 PDF
- 对账单 Excel
- 项目结项报告（成本分析 + 利润汇总）
- 每日/周进度报告（可按模板自动生成）
```

**建议优先级**: 🔴 最高 — 无法导出 = 无法向船东客户交付专业文档，也无法应对审计

## 五、竞争策略

### 5.1 竞品定位分析

| 竞品 | 定位 | 主要客户 | 核心差异化（官网验证） | 弱点（⚠️推断） |
|------|------|---------|----------------------|---------------|
| **DNV ShipManager** | 大型船企全生命周期 | 船东/管理公司 | 模板+知识库复用、集成 PMS | ⚠️ 企业级定价，实施周期长 |
| **SpecTec AMOS** | 大中型船队 | 船东/邮轮 | Spec Builder + 成本控制、AMOS 生态集成 | ⚠️ 系统庞大复杂，上手成本高 |
| **MariApps smartPAL** | 中大型管理公司 | 船东/管理公司 | 缺陷驱动规格书、多币种比价、自动每日报告 | ⚠️ 整体方案偏重，中小团队可能用不完 |
| **VesselMan** | 中型 SaaS | 技术总监/船队 | **AI 成本预测**、数字化检验 (SIRE 2.0)、Condition Manager | ⚠️ 海外产品，无中文支持 |

> ⚠️ 标注的"弱点"和"价格"是基于行业经验推断的，未经直接验证。竞品定价均未在官网公开披露。

### 5.2 LG Management 的核心优势（已有）

1. **贴合第三方服务商实际业务** — 系统由 LG Marine Services 自用驱动开发，天然理解"代表船东监督船厂"的工作流，不是通用软件的生搬硬套
2. **多客户、多船厂项目组合** — 已有客户管理 + 订单/项目管理 + 供应商管理的完整框架，支撑 LG 同时服务多个船东、对接多家船厂
3. **可视化工作流编排** — 竞品均无此功能，审批/流转流程可按项目类型灵活配置
4. **现代化 UI/UX** — Vue 3 + Element Plus，对比传统海事软件体验显著领先
5. **移动端 + 离线上传** — Superintendent 驻场环境网络不稳定，离线拍照上传是刚需
6. **轻量部署** — Docker 一键部署，适合 LG 这样没有大型 IT 部门的服务公司
7. **多币种** — 修船行业天然涉及外币结算，已内置汇率与多币种支持
8. **中文优先** — LG 团队以中文为主要工作语言，国际竞品无中文支持

## 七、对外一页卖点总结

### 对内：为什么 LG Marine Services 需要这套系统？

- **多客户项目组合管理**：LG 同时服务多个船东客户、对接多家船厂，系统提供跨项目的全局视图——哪些项目在进行中、哪些有风险、哪些该结算了
- **工作包级别的范围管控**：把规格书拆成可跟踪的工作包，每一项的进度、验收、变更清清楚楚，不再"只看总进度不知道细节"
- **变更单零遗漏**：每一条变更自动进入审批流，关联合同金额实时更新，不再出现"做了没记、记了没收"——这直接关系到 LG 的项目利润
- **验收证据链可追溯导出**：照片、签字、检查记录按工作包自动归档，一键导出 PDF 验收报告，向船东客户交付时专业规范，审计/争议时不用翻微信找照片
- **结算对账效率提升**：按工作包逐项对账，争议条目可标记跟踪，对账单一键生成，结算周期可显著缩短
- **LG 自身的业务分析**：按客户、船厂、业务线（监造/备件/经纪）的收入与成本分析，帮助管理层看清哪些业务赚钱、哪些客户值得深耕
- **可配置工作流**：不同项目类型的审批流程不同，可视化拖拽配置，不依赖开发修改
- **移动端驻场友好**：Superintendent 在船坞现场用手机拍照取证、更新进度、审批变更，离线也能用

### 对外（如果未来 SaaS 化）：为什么同类第三方服务公司会买？

- **行业场景验证**：系统由真实运营的海事服务公司（LG Marine Services）自用打磨，不是通用 PM 工具的改名包装
- **开箱即用**：内置修理监造项目模板，新项目 5 分钟启动，不需要实施顾问
- **模板化文档输出**：报价单、合同、验收报告、对账单、结项报告均有标准模板，输出专业一致
- **中文优先**：面向中国海事服务市场设计，国际竞品无中文支持
- **轻量部署**：Docker 一键部署，适合没有大型 IT 部门的中小服务公司

## 八、一句话总结

LG Management 是**上海鲁顾船舶服务有限公司（LG Marine Services）的内部项目管理与业务运营系统**，服务于公司的修理监造、备件供应、船舶经纪等多条业务线。当前已具备完整的客户→项目→报价→合同→采购→库存→跟单→结算链路，以及独特的可视化工作流引擎。要形成驻场监造的核心工作闭环，还需补齐 **工作包分解与跟踪**、**变更单/索赔管理**、**验收证据链** 和 **PDF/Excel 导出** 这四个 P0 模块。**规格书构建器**和**项目模板复用**是国际竞品的标准配置，属于 P1 优先补齐项。系统长期价值在于：一方面支撑 LG 自身的多客户、多船厂项目组合管理与业务分析；另一方面，经过实战打磨后，有可能 SaaS 化服务同类第三方海事服务公司。工作流编排引擎是独特的技术壁垒，需要与工作包和变更单系统打通，才能发挥最大价值（如：工作流自动创建工作包、验收通过自动推进节点、变更审批自动触发合同金额更新）。
