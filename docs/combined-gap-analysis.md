## 一、两项目合体后已具备的能力

下表列出 `lg-doc` 为 `LG-management` 补齐的关键能力：

| 能力 | 来源项目 | 实现方式 | 对应竞品分析中的需求项 |
|------|---------|---------|---------------------|
| DOCX/PDF 文档生成 | lg-doc 后端 | `docxtpl` 模板引擎 + LibreOffice 转 PDF + PDF 水印 | P0 #5 报表与文档导出 |
| 照片标注/裁剪/拼图 | lg-doc 前端 | MarkerJS2（标注）+ CropperJS（裁剪）+ Canvas 拼图 | P2 #13 照片标注与绘图 |
| 离线存储 + 同步队列 | lg-doc 前端 | Dexie（IndexedDB）+ WatermelonDB pull/push 协议 | P2 #18 离线增强 |
| AI 辅助填表 | lg-doc 后端 | OpenAI / DeepSeek / Qwen + RAG 历史文档 few-shot | 部分覆盖 AI 能力 |
| 文档模板系统 | lg-doc | 模板定义（schema_json）+ 分组 + 动态字段 + TipTap 富文本 | P1 #9 项目模板（部分） |
| 乐观并发控制 | lg-doc 后端 | `version_int` 字段 + 冲突检测 | 工程质量提升 |
| 审计日志（含 IP/UA） | lg-doc 后端 | `AuditLog` 模型（user_id, action, resource, ip, user_agent） | P1 #11 审计日志 |
| OSS/MinIO 直传 | lg-doc 后端 | presigned URL + multipart 分片上传 | 文件管理增强 |
| Celery 异步任务 | lg-doc 后端 | 文档生成任务 + 清理任务（孤儿照片/过期上传/旧日志） | 后台任务基础 |
| 照片压缩 | lg-doc 后端 | Pillow 自动压缩（按分辨率/质量） | 移动端体验优化 |

### 合体后的完整能力链

```
LG-management 提供:
  客户管理 → 订单/项目管理 → 报价管理 → 合同管理 → 采购/询价 →
  供应商管理 → 备件库 → 库存管理 → 跟单/进度追踪 → 结算(订单级) →
  成本核算 → 工作流编排 → 仪表盘 → RBAC → 移动端(基础)

lg-doc 补齐:
  文档模板定义 → 动态字段填写 → 照片拍摄/标注/裁剪 →
  离线暂存 → 在线同步 → DOCX 生成 → PDF 转换/水印 →
  AI 辅助填表 → 审计日志
```

## 二、仍然缺失的完整清单

### 🔴 P0 — 核心业务闭环（必须做，否则监造工作流无法形成闭环）

#### 1. 工作包分解与跟踪（Work Package Breakdown & Tracking）

- **现状**：两个项目都没有 Work Package 模型
- **LG-management**：跟单用的是 `TrackingNode`（订单级节点），粒度不够
- **lg-doc**：专注文档生成，无工作包概念

```
缺失项:
├── 工作包数据模型（包号、关联项目、所属系统/舱室/设备、工作描述、验收标准）
├── 工作包状态机（未启动 → 进行中 → 待验收 → 已验收 → 关闭，支持返工）
├── 与船厂工单的映射（可选，记录船厂侧工单号便于进度对照）
├── 工量/工时记录（监督方记录实际消耗，或导入船厂数据）
├── 工作包级别成本归集（预算 vs 实际）
└── 工作包列表/详情/筛选 UI（Web + Mobile）
```

**业务影响**：没有工作包粒度的跟踪，"进度透明"无从谈起。Superintendent 到厂后第一件事就是拆工作包。

---

#### 2. 变更单/索赔管理（Variation Order / Change Order）

- **现状**：两个项目完全没有 Variation Order 模型和 API
- **LG-management**：`handleOrderChange` 是订单选择切换，不是变更单

```
缺失项:
├── 变更单数据模型（变更号、关联项目/工作包、发起方、原因分类、金额影响、审批状态）
├── 审批流（监督方确认 → 内部审批 → 船东确认 → 纳入补充协议）
├── 与原合同/报价的关联（合同总金额自动更新）
├── 变更追溯时间线（完整历史：谁发起、谁审批、什么时候确认）
├── 证据附件（照片、船厂报告、检验报告等支撑材料）
└── 变更单 CRUD UI（Web + Mobile 审批）
```

**业务影响**：变更单是关键利润来源。漏记一条变更单 = 直接少收一笔钱。

---

#### 3. 验收记录与证据链（Acceptance & Evidence Trail）

- **现状**：LG-management 有 `FileObjectType.ACCEPTANCE` 枚举但只是空壳；lg-doc 有文档生成能力但没有"验收"业务逻辑

```
缺失项:
├── 验收记录数据模型（关联工作包、检查项清单、检验标准、实际值、合格/不合格）
├── 现场取证增强（照片/视频 + 时间戳 + GPS，移动端拍摄自动附加）
├── 签字确认（监督方签字 + 船厂代表签字）
├── 缺陷清单 Deficiency List（不合格项登记 → 整改跟踪 → 复检关闭）
├── 可追溯导出（按工作包导出完整验收报告 PDF，含所有照片、签字、检查结果）
└── 验收流程 UI（Web 管理 + Mobile 现场操作）
```

**业务影响**：验收证据链是 LG 向船东交付的核心输出，也是争议解决的唯一依据。

---

#### 4. 按工作包结算对账（Settlement Reconciliation Enhancement）

- **现状**：LG-management 有 `Settlement` 模型但只关联 `order_id`，无法按工作包逐项对账

```
缺失项:
├── Settlement 模型扩展（关联工作包维度）
├── 按工作包对账（每个工作包的合同价 vs 实际发生 vs 变更调整）
├── 对账单生成（汇总全部工作包 + 变更单，可利用 lg-doc 的 DOCX 引擎）
├── 争议标记（双方有异议的条目可标记、备注、跟踪解决状态）
└── 发票关联（对账确认后关联发票，跟踪开票/收款状态）
```

**业务影响**：结算效率直接影响 LG 的现金流回收周期。

---

#### 5. 移动端拍照自动关联工作包

- **现状**：LG-management 移动端拍照只能手动选择关联对象（ORDER / TRACKING_NODE 等），没有工作包概念；lg-doc 照片关联到 document_id + group_key

```
缺失项:
├── 移动端进入工作包详情后拍照 → 自动关联该工作包
├── 照片列表按工作包分组查看
├── 照片与验收记录的双向关联
└── 批量拍照模式（连续拍多张自动归入同一工作包）
```

### 🔴 P0 — 现场人员刚需（技术层缺失）

#### 6. 移动端 GPS/经纬度自动采集

- **现状**：两个项目都没有调用 `navigator.geolocation` 或 uni-app 的 `getLocation`

```
缺失项:
├── 拍照时自动获取 GPS 坐标
├── 坐标存入照片元数据（PhotoAsset.meta_json 或新字段）
├── 后端存储经纬度字段
└── 前端地图/位置展示（可选）
```

**业务影响**：GPS 坐标可证明"照片确实是在船厂现场拍的"，增强证据链可信度。

---

#### 7. 移动端拍照时间戳（EXIF 级）

- **现状**：两个项目都没有 EXIF 解析；lg-doc 有 `created_at` 但是上传时间，不是拍照时间

```
缺失项:
├── 前端解析照片 EXIF 提取拍摄时间（可用 exifr 或 exif-js 库）
├── 拍摄时间存入照片元数据
├── 照片详情展示拍摄时间（区别于上传时间）
└── 按拍摄时间排序/筛选
```

**业务影响**：时间戳 + GPS 是验收证据链的基础信息，缺少会影响证据的法律效力。

---

#### 8. 电子签名（E-Signature）

- **现状**：lg-doc 有"签名"字段类型标签但 DynamicField 降级为普通文本输入（未实现手写签名板）；LG-management 完全没有

```
缺失项:
├── 手写签名组件（Canvas 手写板，支持触屏）
├── 签名图片存储（转 PNG/SVG 上传存储）
├── 签名与验收记录关联（谁签的、什么时候签的）
├── 签名防篡改（hash 校验或时间戳签章）
└── 签名在导出 PDF 中正确渲染
```

**业务影响**：验收签字是监督方与船厂确认工作完成的核心动作。

#### 9. 缺陷清单（Deficiency List）

- **现状**：两个项目都没有

```
缺失项:
├── 缺陷数据模型（编号、关联工作包、严重级别、描述、照片、责任方）
├── 缺陷状态机（发现 → 通知整改 → 整改中 → 复检 → 关闭）
```

## 三、前后端字段不一致问题（LG-management 已有 Bug）

这些是 `LG-management` 已有的前后端字段不匹配问题，与 `lg-doc` 无关，但合体后仍需修复：

| 位置 | 前端字段 | 后端字段 | 影响 |
|------|---------|---------|------|
| Settlement | `profit` / `profit_rate` | `gross_profit` / `gross_profit_rate` | 利润显示异常 |
| Cost 页面 | `contract_amount` / `received_amount` | `total_revenue` / `total_received` | 成本页面数据不对 |
| Settlement 状态 | `PENDING` | `PENDING_APPROVAL` | 状态流转可能卡住 |
| Product | `default_price` | 模型无此字段 | 商品页面报错 |
| Supplier | `credit_rating` | 模型无此字段 | 供应商页面报错 |

## 六、建议开发路线图

### Phase A — 基础补丁（1 周）

修复已有问题，打通两项目间的技术基础：

```
├── 修复 5 处前后端字段不一致
```

## 七、一句话总结

`LG-management` 解决了"客户怎么管、报价怎么走、采购怎么跟"，`lg-doc` 解决了"文档怎么生成、照片怎么标注、离线怎么存"——但 **中间这层"驻场监造到底干了什么"的业务闭环（工作包 → 变更 → 验收 → 缺陷 → 结算）在两个项目中都是空白**。GPS/时间戳、电子签名、拍照自动关联等移动端能力也均未实现。这些才是决定系统能否真正帮到现场人员的关键。
