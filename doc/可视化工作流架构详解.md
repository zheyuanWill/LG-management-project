# 可视化工作流编排系统 — 前后端架构详解

## Situation（项目背景）

在船舶物料管理系统中，不同项目类型（监修、备件供应、一般贸易）的业务流程差异很大：监修项目需要经过客户确认、现场勘查、方案审批等多个环节；备件项目可能只需简单的报价→采购→发货流程；而金额超过某个阈值的订单还需要额外的总经理审批。

传统做法是在代码中硬编码每种业务流程，每次新增流程类型都需要改代码、发版本。为了让业务人员能够自主配置流程，我们设计并实现了一套**可视化工作流编排系统**，支持拖拽式流程设计、条件分支、并行网关、自动触发等能力。

---

## Task（技术任务）

设计并实现一个完整的工作流引擎，包含：

1. **前端可视化编辑器** — 基于 ReactFlow 的拖拽式流程图编辑器，支持 10 种节点类型
2. **后端流程引擎** — 基于 FastAPI 的流程执行引擎，支持条件分支、并行网关、安全表达式求值
3. **双向业务联动** — 业务实体状态变更自动驱动流程，流程节点完成自动触发业务动作
4. **完整审计体系** — 每一次状态变迁都有审计日志，支持操作追溯

#### 2.2 编辑器核心 Hook：`useWorkflowEditor`

这个 Hook 是编辑器的"大脑"，封装了所有图编辑状态和操作：

```ts
// components/workflow/useWorkflowEditor.ts

export function useWorkflowEditor() {
  const [nodes, setNodes, onNodesChange] = useNodesState([])
  const [edges, setEdges, onEdgesChange] = useEdgesState([])
  const [selectedNode, setSelectedNode] = useState<Node | null>(null)

  // 历史栈 —— Undo/Redo
  const historyRef = useRef<HistoryEntry[]>([...])
  const historyIndexRef = useRef(0)
  const MAX_HISTORY = 50

  const pushHistory = useCallback(() => {
    const newEntry = {
      nodes: structuredClone(nodes),
      edges: structuredClone(edges),
    }
    const history = historyRef.current.slice(0, historyIndexRef.current + 1)
    history.push(newEntry)
    if (history.length > MAX_HISTORY) history.shift()
    historyRef.current = history
    historyIndexRef.current = history.length - 1
  }, [nodes, edges])
  // ...
}
```

**Undo/Redo 实现要点：**
- 使用 `structuredClone` 做深拷贝，避免引用共享导致的状态污染
- 环形数组结构，最多保留 50 个历史记录
- 每次 `addNode`、`removeSelected`、`onConnect`、`updateNodeData` 操作后自动入栈
- 快捷键绑定：`Ctrl+Z` 撤销、`Ctrl+Shift+Z` / `Ctrl+Y` 重做、`Ctrl+S` 保存

**连接规则校验：**

```ts
const onConnect = useCallback((connection: Connection) => {
  const { source, target } = connection
  if (source === target) return           // 禁止自环
  // ...
  if (sourceType === 'end') return        // 结束节点不能有出边
  if (targetType === 'start') return      // 开始节点不能有入边

  setEdges((eds) => addEdge({ ...connection, id: `e-${Date.now()}` }, eds))
  pushHistory()
}, [nodes, setEdges, pushHistory])
```

**序列化/反序列化 — 前后端字段映射：**

ReactFlow 使用 `sourceHandle`（驼峰），后端 API 使用 `source_handle`（下划线），Hook 在 `getDefinition` 和 `loadDefinition` 时自动做转换：

```ts
// 保存时：前端 → 后端
const getDefinition = useCallback(() => ({
  nodes: nodes.map(n => ({ id: n.id, type: n.type, position: n.position, data: n.data })),
  edges: edges.map(e => ({
    id: e.id, source: e.source, target: e.target,
    source_handle: e.sourceHandle,    // 驼峰 → 下划线
    target_handle: e.targetHandle,
  })),
}), [nodes, edges])

// 加载时：后端 → 前端
const loadDefinition = useCallback((def) => {
  setNodes(def.nodes)
  setEdges(def.edges.map(e => ({
    ...e,
    sourceHandle: e.source_handle,    // 下划线 → 驼峰
    targetHandle: e.target_handle,
  })))
  historyRef.current = [{ nodes: def.nodes, edges: def.edges }]
  historyIndexRef.current = 0
}, [setNodes, setEdges])
```

#### 3.2 安全的条件表达式求值器

条件节点支持自定义表达式，如 `amount > 10000 and project_type == "SUPERVISION"`。为了避免 `eval()` 带来的安全风险，我们实现了基于 Python AST 的安全求值器：

```python
# services/workflow_service.py

_ALLOWED_OPS = {
    ast.Gt: operator.gt,   ast.Lt: operator.lt,
    ast.GtE: operator.ge,  ast.LtE: operator.le,
    ast.Eq: operator.eq,   ast.NotEq: operator.ne,
    ast.Add: operator.add, ast.Sub: operator.sub,
    ast.Mult: operator.mul,
    ast.And: lambda a, b: a and b,
    ast.Or: lambda a, b: a or b,
    ast.Not: operator.not_,
}

def _safe_eval(expr_str: str, context: dict) -> bool:
    tree = ast.parse(expr_str.strip(), mode='eval')

    def _eval_node(node: ast.expr) -> Any:
        if isinstance(node, ast.Constant):    # 字面量: 10000, "SUPERVISION"
            return node.value
        if isinstance(node, ast.Name):        # 变量: amount, status
            if node.id in context:
                return context[node.id]
            raise BusinessError(f"未知变量: {node.id}")
        if isinstance(node, ast.Compare):     # 比较: amount > 10000
            left = _eval_node(node.left)
            for op_node, comp in zip(node.ops, node.comparators):
                op_func = _ALLOWED_OPS.get(type(op_node))
                right = _eval_node(comp)
                if not op_func(left, right):
                    return False
                left = right
            return True
        if isinstance(node, ast.BoolOp):      # 布尔: a and b
            if isinstance(node.op, ast.And):
                return all(_eval_node(v) for v in node.values)
            return any(_eval_node(v) for v in node.values)
        # ... UnaryOp, BinOp, Expression
        raise BusinessError(f"不支持的节点类型: {type(node).__name__}")

    return bool(_eval_node(tree))
```

**安全保证：**
- 不允许函数调用 → 无法执行 `os.system()` 等危险操作
- 不允许属性访问 → 无法读取 `__class__.__bases__` 等内部属性
- 不允许下标操作 → 无法访问 `dict['key']`
- 所有变量必须来自预定义的 `context` → 无法访问任意对象

**上下文变量来自订单数据：**

#### 3.3 图结构验证算法

保存模板前，后端会对流程图进行完整性校验：

```python
# services/workflow_service.py — validate_workflow_definition

def validate_workflow_definition(definition: dict) -> dict:
    nodes = definition.get("nodes", [])
    edges = definition.get("edges", [])
    errors, warnings = [], []

    # 1. 唯一开始/结束节点检查
    start_nodes = [n for n in nodes if n["data"].get("nodeType") == "start"]
    if len(start_nodes) != 1:
        errors.append({"message": "缺少开始节点" or "多个开始节点"})

    # 2. BFS 可达性检查 — 所有节点必须从开始节点可达
    reachable = _bfs(start_nodes[0]["id"], out_edges)
    unreachable = [n for n in nodes if n["id"] not in reachable]
    if unreachable:
        errors.append({"message": f"{len(unreachable)}个节点无法到达", "node_ids": [...]})

    # 3. 拓扑排序检测环 — DAG 合法性
    in_degree = {n["id"]: 0 for n in nodes}
    for e in edges:
        in_degree[e["target"]] += 1
    topo_queue = deque([nid for nid, deg in in_degree.items() if deg == 0])
    visited = 0
    while topo_queue:
        nid = topo_queue.popleft()
        visited += 1
        for edge in out_edges[nid]:
            in_degree[edge["target"]] -= 1
            if in_degree[edge["target"]] == 0:
                topo_queue.append(edge["target"])
    if visited < len(nodes):
        errors.append({"message": "存在循环", "node_ids": cycle_nodes})

    # 4. 条件节点分支完整性（Warning 级别）
    for n in nodes:
        if n["data"].get("nodeType") == "condition":
            handles = {e.get("sourceHandle") for e in out_edges[n["id"]]}
            if "condition-true" not in handles:
                warnings.append({"message": "缺少'是'分支"})
            if "condition-false" not in handles:
                warnings.append({"message": "缺少'否'分支"})

    return {"valid": len(errors) == 0, "errors": errors, "warnings": warnings}
```

校验流程依次执行：**结构完整性 → 可达性 → 无环 → 约束检查**，保证保存的模板一定是可以正确执行的。

#### 3.4 流程推进引擎（状态机）

核心方法 `advance_node` 处理节点状态推进和自动路由：

```python
# services/workflow_service.py — advance_node (核心逻辑)

async def advance_node(self, db, instance_id, node_id, new_status, ...):
    # 1. 权限校验 — 如果节点指定了 assignee，只有该用户能操作
    node_assignee = node_states[node_id].get("assignee")
    if node_assignee and operator.real_name != node_assignee:
        raise ForbiddenError(f"该节点指定由「{node_assignee}」处理")

    # 2. 更新目标节点状态
    node_states[node_id]["status"] = new_status
    node_states[node_id]["completedAt"] = now

    # 3. 自动推进：找到后继节点并根据类型分别处理
    next_node_ids = self._get_next_nodes(node_id, edges)
    for next_id in next_node_ids:
        # 检查所有前驱节点是否都已完成（Join 语义）
        all_preds_done = all(
            node_states[p]["status"] in ("COMPLETED", "SKIPPED")
            for p in predecessors
        )
        if not all_preds_done:
            if next_type == "parallel_gateway" and mode == "join":
                next_state["status"] = "WAITING"  # 等待其他分支
            continue

        # 所有前驱完成，根据节点类型激活
        match next_type:
            case "end":
                next_state["status"] = "COMPLETED"
            case "condition":
                self._activate_condition_node(...)  # 求值并路由
            case "parallel_gateway":
                next_state["status"] = "COMPLETED"
                self._activate_downstream(...)      # 激活所有下游
            case "timer":
                next_state["status"] = "RUNNING"    # 等待定时器
            case _:
                next_state["status"] = "RUNNING"    # 等待人工处理

    # 4. 全局完成检查
    if all(s["status"] in ("COMPLETED", "SKIPPED") for s in node_states.values()):
        instance.status = WorkflowStatus.COMPLETED

    # 5. 触发业务钩子
    await on_node_completed(db, instance, node_id, operator_id)
```

**条件节点的分支路由：**

```python
def _activate_condition_node(self, node_id, node_states, edges, context, now):
    expression = state.get("config", {}).get("condition", "")

    # 求值
    result = _safe_eval(expression, context) if expression else True

    # 标记条件节点本身为 COMPLETED
    state["status"] = "COMPLETED"
    state["condition_result"] = result

    # 激活对应分支的后继节点
    if result:
        targets = self._get_next_nodes_by_handle(node_id, edges, "condition-true")
    else:
        targets = self._get_next_nodes_by_handle(node_id, edges, "condition-false")

    for target_id in targets:
        node_states[target_id]["status"] = "RUNNING"

    # 跳过另一个分支的节点
    other_targets = self._get_next_nodes_by_handle(node_id, edges, other_handle)
    for target_id in other_targets:
        if node_states[target_id]["status"] == "PENDING":
            node_states[target_id]["status"] = "SKIPPED"
```

**并行网关 Fork/Join：**

```
         ┌─ 任务A ─┐
  Fork ──┤          ├── Join ── 下游
         └─ 任务B ─┘
```

- **Fork**：立即标记为 COMPLETED，激活所有下游节点（并行执行）
- **Join**：检查所有前驱是否完成，未完成标记为 WAITING；全部完成后才激活下游

#### 3.5 双向业务联动钩子

这是整个系统最精妙的部分——工作流不是孤立运行的，它与业务实体双向驱动：

```python
# services/workflow_hooks.py

# 方向一：业务 → 工作流
# 当采购单状态变为"已审批"时，自动完成配置了对应触发条件的节点
async def on_entity_status_change(db, entity_type, entity_id, new_status, order_id):
    instances = await _get_running_instances_for_order(db, order_id)
    for instance in instances:
        for node_id, state in instance.node_states.items():
            if state["status"] != "RUNNING":
                continue
            trigger = state.get("config", {}).get("trigger")
            if trigger and trigger["entity"] == entity_type \
               and trigger["status"] == new_status:
                await workflow_service.advance_node(
                    db, instance.id, node_id=node_id,
                    new_status="COMPLETED",
                    notes=f"由{entity_type}#{entity_id}状态变更自动触发",
                )

# 方向二：工作流 → 业务
# 当工作流节点完成后，执行配置的业务动作
async def on_node_completed(db, instance, node_id, operator_id):
    action = state.get("config", {}).get("action")
    if action["type"] == "change_status":
        # 例如：自动将订单状态改为 COMPLETED
        await order_service.update_status(db, instance.order_id, new_status)
    elif action["type"] == "notify":
        await create_notification(db, user_id, title=action["message"])
```

### 五、前后端数据流示例

以一个完整的流程为例：**"金额 > 5 万的订单需要总经理审批"**

```
1. 管理员在编辑器中画流程图：
   开始 → 需求确认 → [金额>50000?] ──是──→ 总经理审批 → 采购执行 → 结束
                                    └──否──→ 采购执行 → 结束

2. 点击"保存"：
   前端: editor.getDefinition() → sourceHandle → source_handle
   POST /workflows/validate → { valid: true }
   POST /workflows/templates → 存入 DB

3. 订单 ORD-2026-001 开始执行：
   auto_create_workflow_for_order() →
   → 查找 project_type 匹配的模板
   → 创建实例，冻结 definition_snapshot
   → 初始化 node_states: { start: COMPLETED, 需求确认: RUNNING, ... }
   → 审计: instance_start

4. 用户在订单详情页完成"需求确认"节点：
   PUT /workflows/instances/{id}/advance { node_id, status: "COMPLETED" }
   → 引擎找到下一个节点: [金额>50000?]
   → _safe_eval("amount > 50000", { amount: 80000, ... }) → True
   → 激活"总经理审批"节点 (RUNNING)，跳过另一条分支 (SKIPPED)
   → 审计: node_advance + condition_evaluate

5. 总经理审批通过：
   → 引擎激活"采购执行" → 激活"结束"
   → 全部 COMPLETED → instance.status = COMPLETED
   → 审计: instance_complete
   → on_node_completed 触发业务动作
```

## Result（技术成果）

1. **灵活性**：业务人员可通过拖拽配置 10 种节点类型的任意组合流程，无需修改代码
2. **安全性**：AST 安全求值器替代 `eval()`，杜绝代码注入；基于 assignee 的节点级权限控制
3. **可靠性**：`definition_snapshot` 冻结机制保证运行中实例不受模板修改影响；完整审计日志支持任意时间点的执行回溯
4. **自动化**：双向业务联动让工作流与业务实体自动驱动，减少人工干预（如采购审批通过 → 自动完成对应节点 → 自动更新订单状态）
5. **工程质量**：前端 Undo/Redo、图结构验证（可达性 + 无环 + 约束）、快捷键支持等保证了良好的编辑体验
